# Фича: Складские операции

## Бизнес-цель

Фиксировать любое изменение остатков как контролируемую операцию, а не как ручное редактирование количества.

## Сценарии

### Приход

- пользователь выбирает склад;
- выбирает один или несколько товаров;
- указывает количество по каждой строке;
- система увеличивает остатки и создает журнал операций.

### Расход

- пользователь выбирает склад;
- система дает выбрать только товары с положительным остатком на этом складе;
- пользователь указывает количество;
- система списывает остаток и создает запись расхода.

### Перемещение

- пользователь выбирает склад-источник и склад-получатель;
- система дает выбрать только товары с остатком на складе-источнике;
- пользователь указывает количество;
- система уменьшает остаток на источнике, увеличивает на получателе и записывает операцию.

## Системная реализация

UI:

- единый `OperationDrawer` для трех типов операций;
- валидация форм на `zod`;
- предварительная загрузка доступного остатка для расхода и перемещения;
- автоочистка/сброс зависимых полей при смене склада.

Backend:

- `create_receipt`;
- `create_sale`;
- `create_transfer`.

После успешной операции приложение выполняет массовое обновление кэша React Query.

## Бизнес-правила

- приход доступен: `warehouse_manager`, `super_admin`;
- расход доступен: `cashier`, `warehouse_manager`, `super_admin`;
- перемещение доступно: `warehouse_manager`, `super_admin`;
- количество всегда должно быть целым и больше 0;
- расход и перемещение не могут превышать доступный остаток;
- перемещение не может выполняться между одинаковыми складами;
- операции возможны только по активным складам и товарам.

## Контроль целостности

- для расхода и перемещения сервер использует блокировку строки остатка `FOR UPDATE`;
- прямой записи в `inventory_balances` из frontend нет;
- журнал операций формируется на стороне БД внутри RPC.

## Ограничения и риски

- комментарий к операции предусмотрен в схеме БД, но в UI сейчас не заполняется;
- нет пакетного расхода и пакетного перемещения по нескольким товарам за один документ;
- расход по фиксированному товару не ограничивает выбор склада только теми, где товар есть, остаток проверяется уже после выбора;
- нет документов-оснований и отмены операции.
