# Фича: Пользователи и администрирование

## Бизнес-цель

Управлять доступом сотрудников к системе через роли и статус активности.

## Пользовательские сценарии

- создать пользователя;
- назначить роль;
- показать логин и пароль один раз после создания;
- сменить пароль существующему пользователю;
- изменить роль пользователя;
- деактивировать пользователя.

## Системная реализация

Создание пользователя:

- выполняется через Edge Function `admin-create-user`;
- сначала создается пользователь в Supabase Auth;
- затем создается профиль в `profiles`.

Смена пароля:

- выполняется через Edge Function `admin-update-user-password`.

Смена роли и деактивация:

- выполняются прямым обновлением `profiles`;
- допустимы только для `super_admin` за счет RLS.

## Бизнес-правила

- раздел доступен только `super_admin`;
- деактивированный пользователь больше не может работать в системе;
- последнего активного `super_admin` нельзя деактивировать через интерфейс;
- пароль нового пользователя показывается один раз сразу после создания;
- минимальная длина пароля в UI и Edge Functions: 6 символов.

## Ограничения и риски

- повторная активация пользователя отдельным сценарием не реализована;
- нет принудительной смены пароля при первом входе;
- нет журналирования административных действий;
- нет групп пользователей и более тонкой permission-модели, только роли.
